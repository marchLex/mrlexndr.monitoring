# Модуль `mrlexndr.monitoring` для 1С-Битрикс

Локальная подсистема мониторинга **базовых показателей** сервера, инфраструктуры и самого сайта на 1С-Битрикс. Модуль периодически выполняет набор проверок (метрик), сохраняет результаты в базе данных, отображает их в административной панели и при необходимости отправляет уведомления (e-mail, Telegram).

**Идентификатор модуля:** `mrlexndr.monitoring`  
**Пространство имён PHP:** `mrLexndr\Monitoring`  
**Вендор / партнёр:** `mrlexndr`

---

## Зачем нужен модуль

- **Проактивный контроль:** до того как пользователи массово увидят проблему, администратор получает сигнал о диске, SSL, лицензии, бэкапах, HTTP и т.д.
- **История в одном месте:** снимки метрик пишутся в таблицу БД; в дашборде видно последний статус, сообщение и время проверки.
- **Расширяемость:** новые проверки и каналы уведомлений можно подключать через события ядра, не меняя ядро модуля.
- **Работа в периметре проекта:** данные остаются на вашем сервере; внешний SaaS-мониторинг не обязателен.

---

## Как устроен модуль (кратко)

1. **Агент** периодически вызывает `Agent::runChecks()` — обходит зарегистрированные «чекеры», учитывает индивидуальный интервал каждой метрики, записывает результат в ORM-таблицу.
2. По правилам настроек решается, **нужно ли отправить уведомление** (смена статуса, восстановление, порог подряд идущих ошибок для защиты от «дребезга»).
3. Второй агент **`clearOldMetrics()`** раз в сутки удаляет записи старше **30 дней**.
4. **Дашборд** в админке показывает последнее состояние каждой метрики; при наличии прав доступна принудительная проверка через AJAX.

Подробности по классам — в PHPDoc в каталоге `lib/mrLexndr/Monitoring/`.

---

## Окружение и требования

| Требование | Описание |
|------------|----------|
| **Платформа** | 1С-Битрикс с поддержкой **D7** (современное ядро, ORM, события). |
| **PHP** | **8.1 и выше** (проверка при установке модуля). |
| **Расширения PHP** | Обязательно: **`curl`**, **`sockets`** (инсталлятор не продолжит установку без них). |
| **Режим агентов** | Рекомендуется запуск агентов по **cron** с установкой константы **`BX_CRONTAB`** в контексте cron — иначе тяжёлые сетевые проверки могут выполняться на хитах и замедлять сайт. На дашборде выводится предупреждение, если cron не используется. |
| **База данных** | Любая СУБД, поддерживаемая вашей редакцией Битрикс (таблица создаётся через ORM при установке). |

Модуль рассчитан на типичный хостинг/VPS с установленным Битрикс и доступом к cron для `cron_events.php` (или эквивалентной схемы запуска агентов у вашей сборки).

## Установка

1. Скопируйте каталог модуля в **`/local/modules/mrlexndr.monitoring/`** (или установите как часть решения с тем же путём относительно корня сайта).
2. В административной части: **Настройки → Управление структурой → Управление модулями** → найдите **`mrlexndr.monitoring`** → **Установить**.
3. После установки в меню появится пункт дашборда (при наличии прав); настройки модуля — в стандартном списке модулей (**Настройки продукта → Настройки модулей → mrlexndr.monitoring**).

При установке создаются:

- таблица **`mrlexndr_monitoring_metrics`** и индекс по `(METRIC_CODE, DATE_CHECK)`;
- почтовое событие **`MRLEXNDR_MONITORING_ALERT`** и шаблон для e-mail;
- файлы в **`/bitrix/admin/`** (точки входа дашборда и AJAX для принудительного запуска);
- два **периодических агента** (см. ниже).

Удаление модуля предлагает опцию **сохранить таблицы БД**; при снятии флага таблица метрик и связанные опции удаляются.

## Встроенные проверки (метрики)

Каждая проверка имеет уникальный **код метрики** (`METRIC_CODE`), название, **интервал по умолчанию** (секунды) и собственные поля на вкладке «Метрики» в настройках. Если для метрики не задан интервал вручную, используется интервал по умолчанию.

| Код | Название | Интервал по умолчанию | Что делает |
|-----|----------|----------------------|------------|
| `DOMAIN_EXPIRE` | Истечение регистрации домена (WHOIS) | 86400 с (1 сутки) | Запрос WHOIS с таймаутами сокета; кэш результата в опциях модуля (снижение нагрузки на регистраторов). |
| `LICENSE_EXPIRE` | Истечение лицензии 1С-Битрикс | 86400 с | Чтение даты окончания лицензии через API ядра (`Application::getLicense()` и т.п., если доступно). |
| `SSL_EXPIRE` | Истечение SSL-сертификата | 86400 с | Подключение TLS к указанному хосту/порту (`stream_socket_client`), учёт **SNI**, разбор сертификата. |
| `BACKUPS` | Резервные копии | 3600 с (1 час) | Поиск файлов бэкапов в каталоге (по умолчанию `…/bitrix/backup`), оценка «свежести» последнего файла. |
| `DISK_SPACE` | Свободное место на диске | 300 с (5 мин) | `disk_free_space` / `disk_total_space` для выбранного пути; порог свободного места в процентах. |
| `TTFB` | Время до первого байта (TTFB) | 300 с | HTTP(S)-запрос через `HttpClient` с таймаутом; пороги WARNING/ERROR в миллисекундах. |
| `HTTP_CODES` | HTTP-коды страниц | 300 с | Набор правил JSON: URL и ожидаемый код ответа; проверка каждого URL. |

Для сетевых операций заданы жёсткие таймауты (в том числе для Telegram в модуле уведомлений). Для HTTP-чекеров дополнительно ограничены редиректы и ослаблена проверка SSL в пользу устойчивости на стендах с самоподписанными сертификатами (см. реализацию в `Ttfb` / `HttpCodes`).

**Расширение:** событие модуля `OnRegisterCheckers` позволяет добавить свои классы, реализующие интерфейс чекера; аналогично `OnRegisterNotifiers` — для каналов уведомлений.

## Переменные и опции по умолчанию

Файл [`default_option.php`](default_option.php) задаёт начальные значения для `\Bitrix\Main\Config\Option` с модулем **`mrlexndr.monitoring`**:

| Ключ опции | Значение по умолчанию | Назначение |
|------------|----------------------|------------|
| `notify_email` | *(пусто)* | E-mail получателя для почтового события `MRLEXNDR_MONITORING_ALERT`. Пусто — письма не отправляются. |
| `notify_telegram_token` | *(пусто)* | Токен Telegram-бота. Пусто — Telegram отключён. |
| `notify_telegram_chat_id` | *(пусто)* | Идентификатор чата Telegram. Пусто — Telegram отключён. |
| `notify_on_error` | `Y` | Отправлять уведомления при статусе **ERROR** (с учётом порога подряд идущих ошибок). |
| `notify_on_warning` | `Y` | Уведомлять при переходе в **WARNING** с другого статуса. |
| `notify_on_ok` | `Y` | Уведомлять при **восстановлении** (переход с ERROR/WARNING на OK). |
| `error_consecutive_threshold` | `2` | Минимум **подряд идущих** результатов со статусом ERROR перед отправкой уведомления об ошибке (антифлад при нестабильной метрике). |
| `consecutive_errors_state` | `{}` | JSON-словарь счётчиков подряд идущих ошибок по кодам метрик; обновляется агентом. |

Дополнительно для каждой метрики в настройках сохраняются (при необходимости):

- `checker_interval_<CODE>` — переопределение интервала в секундах (пусто = брать интервал по умолчанию из чекера);
- `checker_settings_<CODE>` — JSON с полями из формы конкретной проверки (домен, URL, пути, пороги и т.д.).

Отдельные служебные ключи (например, кэш WHOIS для домена) могут добавляться кодом соответствующих чекеров.

## Агенты (расписание)

После установки регистрируются два **периодических** агента модуля:

| Функция | Период вызова агента в Битрикс | Назначение |
|---------|-------------------------------|------------|
| `\mrLexndr\Monitoring\Agent::runChecks();` | **180 секунд** (3 минуты) | Обход всех чекеров с учётом их интервалов и запись результатов. |
| `\mrLexndr\Monitoring\Agent::clearOldMetrics();` | **86400 секунд** (24 часа) | Удаление строк метрик старше **30 дней**. |

Реальный интервал между **одной и той же** метрикой задаётся настройками метрики и последней записью в БД; общий агент может запускаться чаще, но отдельная проверка пропускается, если с последнего снимка прошло меньше своего интервала.

## Уведомления

- **E-mail:** через почтовое событие **`MRLEXNDR_MONITORING_ALERT`** (`EmailNotifier`, при наличии заполненного `notify_email` и настроенной почты на сайте).
- **Telegram:** Bot API через `HttpClient` (`TelegramNotifier`), если заданы токен и chat id.

Ошибки отправки в каналы не ломают сбор других метрик; сбои уведомителей могут фиксироваться в **журнале событий** с типом аудита **`MRLEXNDR_MONITORING_NOTIFIER_ERROR`**.

---

## Административный интерфейс

- **Дашборд** — последний статус по каждой метрике, предупреждение о режиме cron, кнопка принудительного запуска проверок (при достаточных правах).
- **Настройки модуля** — вкладки «Метрики» и «Уведомления».

Права доступа к функционалу задаются через матрицу модуля в файле [`access.php`](access.php) и групповые права на модуль `mrlexndr.monitoring` в Битрикс.

---

## Автозагрузка классов

Классы расположены по PSR-4 в **`lib/mrLexndr/Monitoring/`**. Соответствие пространства имён и путей описано в [**`.settings.php`**](.settings.php). Файл [**`include.php`**](include.php) содержит только защиту от прямого доступа по HTTP (`B_PROLOG_INCLUDED`) — без ручной регистрации классов в этом файле.

## Устранение неполадок

| Симптом | Что проверить |
|---------|----------------|
| Метрики не обновляются | Настроен ли cron для агентов, выполняется ли `Agent::runChecks`, нет ли ошибок в логах PHP/Битрикс. |
| Дашборд пустой | Дождаться первого успешного прохода агента или запустить проверку принудительно с правами администратора/настройки. |
| Нет писем | Заполнен ли `notify_email`, создан ли шаблон события `MRLEXNDR_MONITORING_ALERT`, работает ли почта сайта. |
| Нет Telegram | Токен и chat id, доступ сервера к `api.telegram.org`, записи в журнале с `MRLEXNDR_MONITORING_NOTIFIER_ERROR`. |
| Установка не проходит | PHP ≥ 8.1, расширения `curl` и `sockets`. |

## Лицензия и поддержка

Условия распространения и поддержки определяются автором модуля / партнёром **mrlexndr**. При использовании в продакшене рекомендуется протестировать модуль на копии проекта и настроить мониторинг cron до включения строгих оповещений.
